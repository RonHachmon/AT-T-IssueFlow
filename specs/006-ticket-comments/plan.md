# Implementation Plan: Ticket Comments

**Branch**: `006-ticket-comments` | **Date**: 2026-05-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/006-ticket-comments/spec.md`

## Summary

Allow authenticated users to post, list, edit, and delete plain-text comments on any ticket.
Comments are hard-deleted, carry a JPA `@Version` optimistic-lock counter (same pattern as tickets),
and expose four endpoints nested under `/tickets/{ticketId}/comments`.
The implementation adds a new `comment` package, a `StaleVersionException`, a Flyway migration,
and enables JPA auditing globally.

## Technical Context

**Language/Version**: Java 21 + Spring Boot 3.x (per `pom.xml`)
**Primary Dependencies**: Spring Data JPA, Spring Security (JWT), Lombok, MapStruct, springdoc-openapi, Bean Validation, Zalando Problem Details

**Target Platform**: Linux/macOS server, single-node
**Project Type**: REST web service
**Performance Goals**: < 200 ms response under normal (single-node) load
**Constraints**: All endpoints require JWT bearer authentication; optimistic locking enforced on edits
**Scale/Scope**: Same as existing features; no additional scale requirements

## Complexity Tracking

| Deviation | Why Accepted | Simpler Alternative Rejected Because |
|-----------|--------------|--------------------------------------|
| All endpoints return `200 OK` (not `201`/`204`) | Project-wide README convention; deviating would create inconsistency in the API surface visible to clients. | Following standard REST conventions (`201`, `204`) would break the documented contract pattern every other endpoint in the project uses. |
| `@EnableJpaAuditing` added (new mechanism alongside existing `@PrePersist`/`@PreUpdate`) | User explicitly requested `@CreatedDate`/`@LastModifiedDate` for the Comment entity. Two mechanisms coexist; existing entities are unaffected. | Continuing with `@PrePersist`/`@PreUpdate` on Comment would be simpler but violates the explicit implementation direction. |

## Implementation Notes

### Comment Entity

```java
@Entity
@Table(name = "comments")
@EntityListeners(AuditingEntityListener.class)
@Getter @Setter @NoArgsConstructor
public class Comment {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "content", columnDefinition = "TEXT", nullable = false)
    private String content;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "ticket_id", nullable = false, updatable = false)
    private Ticket ticket;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "author_id", nullable = false, updatable = false)
    private User author;

    @CreatedDate
    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    @Column(name = "updated_at", nullable = false)
    private Instant updatedAt;

    @Version
    @Column(name = "version", nullable = false)
    private Long version;
}
```

### Service Contract

```java
public interface CommentService {
    CommentResponse createComment(Long ticketId, CreateCommentRequest request, String authorUsername);
    PagedResponse<CommentResponse> listComments(Long ticketId, Pageable pageable);
    CommentResponse updateComment(Long ticketId, Long commentId, UpdateCommentRequest request);
    void deleteComment(Long ticketId, Long commentId);
}
```

- `createComment`: validates ticket active, loads author by username, maps request → entity, saves.
- `listComments`: validates ticket active, returns `Page<Comment>` mapped to `PagedResponse<CommentResponse>`.
- `updateComment`: loads comment, verifies comment belongs to `ticketId` (or relies on path binding), checks `request.version() == comment.getVersion()` → `StaleVersionException` if stale, updates content, saves.
- `deleteComment`: loads comment by ID (and `ticketId` for path correctness), calls `repository.delete(comment)`.

### StaleVersionException

```java
public class StaleVersionException extends RuntimeException {
    public StaleVersionException() {
        super("Resource has been modified by another request; re-fetch and retry.");
    }
}
```

`GlobalExceptionHandler` maps it to `409 Conflict` via `ProblemDetailFactory`.

### Unit Tests (`CommentServiceTest`)

| Test | Scenario |
|------|----------|
| `createsCommentSuccessfully` | Happy-path: ticket active, author found, comment saved |
| `throwsNotFoundWhenTicketAbsentOnCreate` | Ticket does not exist or is soft-deleted |
| `throwsNotFoundWhenAuthorNotFound` | Username not in the database |
| `listsCommentsOrderedByCreatedAt` | Repo returns multiple comments; service returns them in order |
| `throwsNotFoundWhenTicketAbsentOnList` | Ticket does not exist on list call |
| `updatesCommentSuccessfully` | Happy-path: version matches, content updated, version incremented |
| `throwsStaleVersionExceptionOnVersionMismatch` | request.version differs from stored version → StaleVersionException |
| `throwsNotFoundWhenCommentAbsentOnUpdate` | Comment ID not in DB |
| `deletesCommentSuccessfully` | Comment exists → repository.delete called |
| `throwsNotFoundWhenCommentAbsentOnDelete` | Comment ID not in DB |

### Postman Collection (`comments.postman_collection.json`)

Covers:
- Auth setup (re-uses existing environment `authToken` variable)
- `POST /tickets/{{ticketId}}/comments` — happy path, blank content, missing ticket
- `GET /tickets/{{ticketId}}/comments` — list with results, empty list, deleted ticket
- `PATCH /tickets/{{ticketId}}/comments/{{commentId}}` — happy path, stale version, missing comment
- `DELETE /tickets/{{ticketId}}/comments/{{commentId}}` — happy path, missing comment
